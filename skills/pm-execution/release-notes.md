# 릴리즈 노트

> 기술적 변경사항을 고객/바이어 대면 릴리즈 노트로 변환 — pm-skills 원본 프레임워크를 FPOF 패션 하우스 컨텍스트에 맞게 커스터마이징

## 언제 사용
- 시즌 런칭 시 바이어/내부용 릴리즈 노트 작성
- 신상품 출시 공지문 작성
- 시즌 변경사항 요약 문서 작성

## FPOF 컨텍스트
- 패션 시즌 런칭, 컬렉션 업데이트, 채널별 상품 변경 등의 릴리즈 노트
- `tone-manner.json`의 브랜드 보이스 준수 (고객 대면 콘텐츠)
- `channels.json`의 채널별 톤 차이 반영 (바이어용 vs 소비자용)

## 사전 준비
- 변경사항 원본 (상품 기획서, 라인시트 변경, 마케팅 캘린더 등)
- `tone-manner.json` 로드
- 대상 독자 확인 (바이어, 내부, 소비자)

## 실행 절차

### Step 1: 원본 자료 수집
제공된 파일(상품 기획서, 라인시트, 변경 로그)을 읽고 추출:
- 무엇이 변경되었는가 (신상품, 개선, 수정)
- 누구에게 영향을 미치는가 (어떤 고객/바이어 세그먼트)
- 왜 중요한가 (사용자 이점)

### Step 2: 변경사항 분류
- **신규 (New)**: 완전히 새로운 상품/라인/기능
- **개선 (Improvements)**: 기존 상품/서비스의 향상
- **수정 (Fixes)**: 해결된 이슈
- **주요 변경 (Breaking Changes)**: 바이어/고객 조치가 필요한 사항
- **종료 (Deprecations)**: 단종 예정 상품

### Step 3: 각 항목 작성
- 기술적 변경이 아닌 사용자 이점으로 리드
- 평이한 언어 사용 — 내부 코드명, 번호 회피
- 항목당 1-3문장
- 사용자 제공 시 비주얼/이미지 포함

**변환 예시**:
- 내부: "26SS 그래픽 티 라인 IP 캐릭터 키키 적용 프린트 3종 추가"
- 대외: "키키와 함께하는 새로운 그래픽 티 3종이 추가되었습니다. 키치한 스트리트 감성을 일상에서 자유롭게 표현해보세요."

### Step 4: 릴리즈 노트 구조화

```
# [브랜드/시즌] — [버전/날짜]

## 신규 상품
- **[상품명]**: [설명 및 가치]

## 개선사항
- **[영역]**: [무엇이 개선되었고 어떻게 도움이 되는지]

## 수정사항
- [사용자 언어로 이슈 설명]

## 주요 변경 (해당 시)
- **조치 필요**: [바이어/고객이 해야 할 일]
```

### Step 5: 톤 조정
- 바이어용: 전문적이고 데이터 중심
- 소비자용: `tone-manner.json` 기반 브랜드 보이스
- 내부용: 간결하고 실무 중심

### Step 6: 와키윌리 적용 필터링
- `tone-manner.json`의 브랜드 어휘/톤 반영
- IP 캐릭터 활용 상품의 경우 `ip-bible.json` 참조하여 세계관 일관성 확인

## 산출물 포맷
```markdown
# 와키윌리 [시즌] 릴리즈 노트 — [날짜]

## 신규 상품
## 개선사항
## 수정사항
## 주요 변경
```

## 완료 조건
- 모든 변경사항이 분류되어 포함
- 사용자 이점 중심 작성
- 대상 독자에 맞는 톤 적용
- 브랜드 보이스 준수
