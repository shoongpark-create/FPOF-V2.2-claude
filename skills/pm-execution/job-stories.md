# 잡 스토리 작성

> 고객 관점의 상품/서비스 요구사항을 'When [상황], I want to [동기], so I can [결과]' 포맷으로 정의 — pm-skills 원본 프레임워크를 FPOF 패션 하우스 컨텍스트에 맞게 커스터마이징

## 언제 사용
- 고객 상황과 동기 중심의 요구사항 정의 시
- JTBD(Jobs-to-be-Done) 프레임워크로 백로그 아이템 작성 시
- 자사몰/앱 기능 기획 시 고객 관점 스토리 도출 시
- 역할이 아닌 상황에 초점을 맞춘 스토리 작성 시

## FPOF 컨텍스트
- `personas.json`의 UNI/WOMAN 페르소나를 잡 스토리의 상황과 동기에 반영
- 와키윌리 고객(18~25세 트렌드리더)의 구매 여정과 라이프스타일을 상황 설정에 활용
- 자사몰, 무신사 등 채널별 고객 경험 차이를 반영

## 사전 준비
- `.fpof-state.json`에서 현재 시즌/단계 확인
- `personas.json`에서 페르소나별 특성, 행동 패턴, 니즈 확인
- 사용자가 제공하는 기능 설명, 디자인 파일, 고객 리서치 읽기

## 실행 절차

### Step 1: 사용자 상황 식별
니즈를 촉발하는 사용자의 상황(situation)을 파악한다.

### Step 2: 동기 정의
사용자 행동의 근본 동기(motivation)를 정의한다.

### Step 3: 결과 명확화
사용자가 달성하고자 하는 결과(outcome)를 명확히 한다.

### Step 4: JTBD 프레임워크 적용
역할이 아닌 해야 할 일(job)에 초점을 맞춘다.

### Step 5: 수용 기준 작성
결과가 달성되었는지 검증하는 수용 기준을 작성한다.

### Step 6: 관찰 가능하고 측정 가능한 언어 사용

### Step 7: 디자인 목업/프로토타입 연결

### Step 8: 와키윌리 적용 필터링
- 상황이 `personas.json`의 페르소나 행동 패턴과 일치하는지 확인
- 동기가 와키윌리 브랜드 경험(키치 스트릿, IP 유니버스)과 정합하는지 확인
- 결과가 5대 경영목표 중 하나에 기여하는지 점검

## 스토리 템플릿
```markdown
**제목:** [잡 결과 또는 성과]

**설명:** When [상황], I want to [동기], so I can [결과].

**디자인:** [디자인 파일 링크]

**수용 기준:**
1. [상황이 적절히 인식됨]
2. [시스템이 원하는 동기를 가능하게 함]
3. [진행 상황 또는 피드백이 보임]
4. [결과가 효율적으로 달성됨]
5. [엣지 케이스가 적절히 처리됨]
6. [통합 및 알림이 작동함]
```

## 예시 (와키윌리 컨텍스트)
```markdown
**제목:** 신상품 입고 알림 확인

**설명:** When 내가 좋아하는 캐릭터(키키) 콜라보 신상품이 나올 것 같을 때(상황),
빠르게 입고 알림을 설정하고 싶어서(동기),
품절 전에 확실하게 구매할 수 있다(결과).

**디자인:** [Figma 링크]

**수용 기준:**
1. 상품 상세 페이지에 '입고 알림' 버튼이 노출됨
2. 알림 설정 시 앱 푸시 또는 카카오톡 중 선택 가능
3. 입고 시 5분 이내 알림 발송
4. 알림에서 바로 구매 페이지로 이동 가능
5. 이미 품절된 경우 '재입고 알림'으로 전환
6. 알림 이력을 마이페이지에서 확인 가능
```

## 완료 조건
- 기능에 대한 완전한 잡 스토리 세트 제시
- 모든 스토리가 'When...I want...so I can' 포맷 준수
- 각 스토리에 6~8개의 결과 중심 수용 기준 포함
- 상황과 동기가 사용자 관점에서 작성됨

## 체크리스트

- [ ] 동기가 기능이 아닌 근본 니즈를 반영하는가?
- [ ] 수용 기준이 결과 중심으로 작성되었는가?
- [ ] `personas.json`의 페르소나와 연결되었는가?
- [ ] 디자인 파일 링크가 포함되었는가?
