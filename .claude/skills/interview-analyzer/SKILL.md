# Interview Analyzer (면접 분석 및 채용 적합성 평가)

면접 녹취록·메모를 **구조화된 면접 분석 보고서**로 변환하고, 조직 문서 기반 **인재상 비교 분석**까지 수행하는 스킬.

## 작업 모드

요청에 따라 두 가지 모드로 동작:

| 모드 | 트리거 | 산출물 |
|------|--------|--------|
| **A. 면접 정리** | "면접 정리해줘", "면접내용 정리" | 구조화된 면접 답변 정리 |
| **B. 인재상 비교 분석** | "인재상 비교", "채용 적합성 평가", "적절한 인재인지 판단" | 조직 요건 vs 후보자 역량 매칭 + 채용 권고 |

모드 B는 모드 A 완료 후 추가 요청 시, 또는 처음부터 함께 요청 시 수행.

---

## 모드 A: 면접 정리

### 1단계: 소스 분석

1. 현재 노트 Read → 원본 내용 파악
2. 콘텐츠 유형 판별:
	- **녹취록 (STT)**: 구어체, 화자 구분 없음, 오탈자 다수 → 맥락 추론 필요
	- **메모**: 키워드 중심, 불완전 문장 → 의미 보완 필요
	- **정리된 노트**: 이미 구조화 → 보강/재구성만 수행

### 2단계: YAML 프론트매터

```yaml
---
type: meeting
aliases:
  - "{후보자명} 면접"
  - "{브랜드/팀} {포지션} 면접"
author:
  - "[[Sherman]]"
date created: {YYYY-MM-DD}
tags:
  - 면접
  - {브랜드명}
  - {직무}
  - 채용
---
```

### 3단계: 후보자 정보 테이블

면접 내용에서 추출하여 작성:

```markdown
## 후보자 정보

| 항목 | 내용 |
|------|------|
| **이름** | {이름} |
| **출생** | {연도}년생 (만 {나이}세) |
| **지원 포지션** | {포지션명} |
| **현 직장** | {회사} - {직책} ({입사시기}~) |
| **주요 경력** | {핵심 경력 요약} |
| **특이사항** | {이직 빈도, 특기 등} |
```

추론이 필요한 항목은 면접 내용에서 간접 정보 수집. 확인 불가 시 "미확인" 표기.

### 4단계: 질문-답변 구조화

기존 질문 목록이 있으면 그대로 유지. 없으면 면접 흐름에서 질문을 추출.

각 질문에 대해:
1. **핵심 답변** 요약 (구어체 → 문어체 전환)
2. **구체적 수치/사례** 강조 (볼드 처리)
3. **면접관 코멘트** 있으면 인용 블록으로 분리
4. STT 오탈자·음성 인식 오류는 맥락에서 추론하여 교정
	- 교정 확신도 낮으면 원문 병기

**답변 정리 원칙:**
- 사실(Fact)과 의견(Opinion) 분리
- 정량적 성과는 반드시 수치로 표기 (매출 4.7억, 유입 400명 등)
- 실패 경험에서의 교훈은 ❌/✅ 아이콘으로 구분
- 면접관 반응·코멘트는 `> **🔍 면접관 코멘트**:` 형식

### 5단계: 종합 평가 테이블

```markdown
## 종합 평가 포인트

| 평가 항목 | 강점 | 우려 |
|-----------|------|------|
| **{항목}** | {강점} | {우려사항 또는 —} |
```

평가 항목 기본 카테고리:
- 콘텐츠/직무 감각, AI 활용, 실행력, 데이터 역량, 리더십, 조직 적응, 안정성, 포지션 핏

### 6단계: 원본 보존

원본 메모/노트 내용은 `## 원본내용` 섹션으로 이동하여 보존.

## 모드 B: 인재상 비교 분석

### 1단계: 조직 문서 탐색

면접 노트의 상위 폴더 및 관련 폴더에서 조직 컨텍스트 문서를 탐색:

```
탐색 대상 (우선순위):
1. 같은 폴더 내 문서 (업무 가이드, 보고서 등)
2. 상위 폴더 (사업부 방향성, 신년사 등)
3. 관련 하위 폴더 (1on1, 회의록 등)
```

**핵심 탐색 키워드:**
- 신년사, 비전, 방향성, 핵심 역량, AI, AX, DX
- 인재상, 업무 가이드, 사업 계획, 워크샵
- 원온원, 조직 진단, KPI

### 2단계: 인재상 도출

조직 문서에서 **인재 요건을 6대 핵심 요건**으로 구조화:

```markdown
### {조직명}이 추구하는 인재상 (6대 핵심 요건)

| # | 핵심 요건 | 출처 근거 |
|---|----------|----------|
| 1 | **{요건명}** — {설명} | {출처 문서명, 핵심 인용} |
```

도출 원칙:
- 반드시 **출처 문서명과 근거**를 명시
- 여러 문서에서 반복 언급되는 요건에 높은 가중치
- 면접관이 면접 중 강조한 니즈도 반영
- 조직의 **현재 시급 과제**에서 역추론한 요건 포함

### 3단계: 역량 매칭 분석

```markdown
### 후보자 역량 매칭 분석

| 조직 요건 | 후보자 수준 | 판정 | 근거 |
|-----------|-----------|------|------|
| **① {요건}** | ⭐~⭐⭐⭐⭐⭐ (5점) | 🔴/🟡/🟢 | {면접 답변 기반 근거} |
```

판정 기준:
- 🟢 **충족** (4-5점): 요건에 부합하는 구체적 경험·성과 있음
- 🟡 **부분 충족** (3점): 관련 경험은 있으나 수준/깊이가 부족하거나 검증 필요
- 🔴 **갭 큼** (1-2점): 해당 역량의 경험·사례가 없거나 매우 부족

### 4단계: 조직 상황 적합성

현재 조직의 **시급 과제** (원온원, 조직 진단 등에서 도출)와 후보자의 적합도:

```markdown
### 조직 상황과의 적합성 분석

| 시급 과제 | 필요한 역할 | 후보자 적합도 | 판단 근거 |
|----------|-----------|-------------|----------|
```

### 5단계: 최종 판단

**결론 형식:**

```markdown
#### 결론: {🟢/🟡/🔴} **{판정}** — {부제}
```

판정 옵션:
- 🟢 **적합 (Strong Fit)** — 핵심 요건 대부분 충족
- 🟡 **조건부 적합 (Conditional Fit)** — 역할 재정의 또는 보완 조건 필요
- 🔴 **부적합 (Poor Fit)** — 핵심 요건과 구조적 미스매치

**필수 포함 항목:**

| 구분 | 내용 |
|------|------|
| **채용 시 기대할 수 있는 것** | {구체적 나열} |
| **채용 시 기대하기 어려운 것** | {구체적 나열} |
| **핵심 리스크** | ❶ ❷ ❸ 번호로 나열 |

### 6단계: 채용 의사결정 옵션

3가지 옵션(채용/보류/불채용)을 조건·장단점과 함께 제시:

```markdown
#### 채용 의사결정 옵션

| 옵션 | 조건 | 장점 | 단점 |
|------|------|------|------|
| **A. 채용 (역할 재정의)** | {조건} | {장점} | {단점} |
| **B. 보류** | {조건} | {장점} | {단점} |
| **C. 불채용** | {조건} | {장점} | {단점} |
```

마지막에 `> **💡 권고**:` 블록으로 현실적 권고안 제시. 조직의 긴급도(공석 기간, 시즌 등)를 고려.

## STT 녹취록 처리 가이드

면접 녹취록은 음성 인식(STT) 특유의 오류가 많으므로 다음 규칙 적용:

| 오류 유형 | 예시 | 처리 |
|----------|------|------|
| 동음이의어 | "세정" ↔ "세장" | 맥락에서 추론하여 교정 |
| 브랜드명 오인식 | "디디도브" → "디디돋고" | 정확한 명칭으로 교정 |
| 문장 끊김 | 불완전 문장 | 전후 맥락으로 보완 |
| 화자 구분 없음 | 면접관/후보자 뒤섞임 | 질문 패턴, 경어 사용으로 화자 추론 |
| 외국어 혼입 | 영어/일어 음차 | 원래 단어로 복원 |

## 출력 포맷 규칙

- **들여쓰기**: TAB 사용 (스페이스 아님)
- **내부 링크**: `[[wikilink]]` 형식
- **YAML 내 wikilink**: 반드시 쌍따옴표로 감싸기
- **날짜**: ISO 8601 (YYYY-MM-DD)
- **테이블**: Obsidian 호환 마크다운 테이블
- **이모지**: 판정 아이콘(🔴🟡🟢), 별점(⭐), 구분 아이콘(❌✅❶❷❸)만 사용
