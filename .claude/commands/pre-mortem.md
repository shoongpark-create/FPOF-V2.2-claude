# 프리모템 리스크 분석 (Pre-Mortem)

런칭이 실패했다고 가정하고 역추적하여 리스크를 사전에 식별합니다. Tigers/Paper Tigers/Elephants 프레임워크를 적용합니다.

## 사용법
`/pre-mortem <PRD, 런칭 계획, 또는 기능 설명>` — 텍스트, 파일, 기존 산출물 모두 지원합니다.

## 절차

### Step 1: 계획 수집
1. `.fpof-state.json`에서 현재 시즌과 단계 확인
2. 입력 수용: PRD, 기능 스펙, 런칭 계획, 프로젝트 브리프, 구두 설명
3. 상세할수록 리스크 분석이 정밀해짐

### Step 2: 리스크 식별
`skills/quality/quality-gate.md` (Pre-Mortem 내장) 참조:
런칭이 실패했다고 가정하고 카테고리별 리스크 생성:
- **기술적**: 성능, 확장성, 통합 실패, 데이터 이슈
- **사용자**: 도입 장벽, 사용성 문제, 기대 불일치
- **비즈니스**: 매출 영향, 경쟁 대응, 시장 타이밍
- **운영**: CS 부하, 문서 공백, 교육 필요
- **의존성**: 외부 서비스, 팀 간 핸드오프, 규제

### Step 3: 리스크 분류

**Tigers** — 실패를 유발할 수 있는 실질적 리스크:
- 런칭 차단(Launch-blocking) / 빠른 후속(Fast-follow) / 추적(Track)

**Paper Tigers** — 무섭게 보이지만 관리 가능한 리스크:
- 왜 관리 가능한지 설명
- 진짜 Tiger가 되려면 무엇이 변해야 하는지

**Elephants** — 팀이 알지만 회피하는 리스크:
- 정치적, 조직적, 불편한 리스크를 건설적으로 제시

### Step 4: Go/No-Go 체크리스트 포함한 보고서 작성

### Step 5: 산출물 저장
산출물을 `output/[시즌]/[프로젝트]/plan_pre-mortem.md`에 저장한다.
`.fpof-state.json`의 artifacts 배열에 파일 경로 추가.

### Step 6: 다음 단계 제안
- "리스크 완화를 **PRD에 반영**하시겠어요?"
- "가장 위험한 영역에 대한 **테스트 시나리오**를 만드시겠어요?"
- "이 결과로 **런칭 체크리스트**를 작성하시겠어요?"
