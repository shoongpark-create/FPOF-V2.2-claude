# 세션 5: 나만의 AI 스킬 만들기 (Skill Creator)

> **소요시간**: 40분 (설명 15분 + 데모 10분 + 실습 15분)
> **대상**: AI 활용 업무를 자동화하고 싶은 멤버
> **목표**: Claude Code 스킬의 구조를 이해하고, 실무에서 반복하는 작업을 스킬로 만드는 과정을 체험

---

## 1. 스킬이란?

### 핵심 개념

스킬은 Claude에게 **전문 지식과 작업 절차를 가르치는 문서**입니다.

| 비유 | 설명 |
|------|------|
| 스킬 = **업무 매뉴얼** | 신입사원에게 주는 "이렇게 하면 됩니다" 가이드 |
| SKILL.md = **표준 작업 지침서(SOP)** | 언제, 무엇을, 어떻게, 어떤 순서로 할지 적은 문서 |
| 트리거 = **업무 분장** | "이런 요청이 오면 이 매뉴얼대로 처리해" |

### 스킬이 없을 때 vs 있을 때

```
[스킬 없음]
나: "트렌드 브리프 작성해줘"
AI: (일반적인 지식으로 대충 작성 — 브랜드 맥락 없음)

[스킬 있음]
나: "트렌드 브리프 작성해줘"
AI: (FPOF 트렌드 브리프 스킬 로딩)
    → 와키윌리 brand.config.json 참조
    → BTA 가이드라인 준수
    → 5대 경영목표 연계
    → 표준 포맷으로 산출물 생성
```

### FPOF에 이미 있는 스킬 예시

| 스킬 | 하는 일 | 스킬이 없었다면? |
|------|---------|---------------|
| `trend-research` | 경쟁사 프로파일링, TAM/SAM/SOM 추정 포함 트렌드 분석 | 일반적인 패션 트렌드 글만 작성 |
| `copywriting` | 와키윌리 톤앤매너·채널별 톤·브랜드 어휘 준수 카피 | 브랜드 보이스 없는 generic 카피 |
| `pinterest-crawl` | 5단어 규칙, 키워드 자동 폴더 분류 | 수동 저장 또는 제대로 안 되는 크롤링 |
| `costing-ve` | 가격 포지셔닝, 경쟁 벤치마크 포함 원가 분석 | 단순 마진 계산만 |

## 2. 스킬의 구조

### 파일 구조

```
my-skill/
├── SKILL.md              ← 필수. AI가 읽는 메인 문서
├── scripts/              ← 선택. 자동 실행 스크립트
│   └── generate.py
├── references/           ← 선택. 필요할 때 읽는 참고 문서
│   ├── guide-a.md
│   └── guide-b.md
└── assets/               ← 선택. 템플릿, 아이콘 등
    └── template.docx
```

### SKILL.md 구성

```markdown
---
name: my-skill                    ← 스킬 이름 (영문, 하이픈 구분)
description: >                    ← 트리거 조건 (이게 제일 중요!)
  이 스킬은 ~할 때 사용합니다.
  ~라는 요청이 오면 반드시 이 스킬을 사용하세요.
---

# 스킬 제목

## 언제 사용하나?
- 이런 상황에서 사용
- 저런 요청이 올 때 사용

## 작업 절차
1. 첫 번째 단계
2. 두 번째 단계
3. 세 번째 단계

## 출력 형식
이런 포맷으로 결과를 만들어라

## 예시
Input: ~
Output: ~
```

### 3단계 로딩 시스템

| 단계 | 언제 로딩? | 크기 |
|------|-----------|------|
| **메타데이터** (name + description) | 항상 (모든 대화) | ~100단어 |
| **SKILL.md 본문** | 스킬이 트리거될 때 | ~500줄 이내 권장 |
| **references/ 파일** | SKILL.md가 "이 파일을 읽어라"고 할 때 | 무제한 |

> **핵심**: description이 좋아야 스킬이 제때 작동합니다. 너무 짧으면 트리거 안 됨, 너무 넓으면 불필요할 때도 작동.

## 3. 스킬 만들기 5단계

### 단계 1: 의도 정리

**4가지 질문에 답하세요:**

| # | 질문 | 예시 (주간 리뷰 스킬) |
|---|------|---------------------|
| 1 | 이 스킬로 AI가 뭘 하게 할 건가? | 매출 데이터 엑셀을 읽고 경영진용 주간 리뷰 요약을 만든다 |
| 2 | 어떤 말/상황에서 작동해야 하나? | "주간 리뷰 만들어줘", "이번 주 실적 정리해줘" |
| 3 | 결과물 형태는? | 마크다운 요약 + HTML 대시보드 |
| 4 | 테스트는 필요한가? | O (데이터 정확성 검증 필요) |

### 단계 2: 리서치 & 인터뷰

스킬을 만들기 전에 명확히 해야 할 것들:

- **엣지 케이스**: 데이터가 비어 있으면? 컬럼명이 다르면?
- **입력 형식**: 어떤 파일을 받는지? (xlsx, csv, json...)
- **성공 기준**: 어떤 결과가 나와야 "잘됐다"인지?
- **기존 참고**: 이미 비슷한 스킬이 있는지?

### 단계 3: SKILL.md 작성

**작성 원칙:**

| 원칙 | 설명 | 예시 |
|------|------|------|
| **Why 설명** | "왜 이렇게 하는지" 이유를 적는다 | ❌ "반드시 BTA를 넣어라" → ✅ "BTA를 넣는 이유는 전사 가이드라인에서 모든 상품 기획에 BTA 구성을 필수로 정했기 때문이다" |
| **예시 포함** | 실제 Input/Output 예시를 넣는다 | Input: "이번 주 매출 정리해줘" → Output: 표준 포맷 마크다운 |
| **간결하게** | 500줄 이내. 길면 references/로 분리 | 본문 = 절차, references = 상세 가이드 |
| **강제보다 설득** | MUST/NEVER보다 이유 설명이 효과적 | "브랜드 보이스가 일관되어야 고객이 와키윌리를 기억하므로, tone-manner.json을 참조한다" |

### 단계 4: 테스트

2~3개 현실적인 테스트 프롬프트를 작성하고 실행합니다.

```json
{
  "skill_name": "weekly-review",
  "evals": [
    {
      "id": 1,
      "prompt": "이번 주 매출 데이터 정리해줘. 엑셀은 workspace/26SS/weekly/data/에 있어.",
      "expected_output": "경영진용 주간 리뷰 마크다운 + 대시보드 HTML"
    },
    {
      "id": 2,
      "prompt": "지난주 대비 이번 주 실적 비교해줘",
      "expected_output": "WoW 비교 테이블 + 인사이트 요약"
    }
  ]
}
```

### 단계 5: 반복 개선

```
테스트 실행 → 결과 확인 → 피드백 → 스킬 수정 → 다시 테스트 → ...
```

이 루프를 결과가 만족스러울 때까지 반복합니다.

## 4. 실습: 간단한 스킬 직접 만들기

### 실습 과제: "일일 체크인" 스킬

매일 아침 팀원들이 "오늘 할 일 정리해줘"라고 하면, 표준 형식으로 일일 체크인 문서를 만드는 스킬입니다.

#### Step 1: 폴더 생성

```bash
mkdir -p system/skills/practice/daily-checkin
```

#### Step 2: SKILL.md 작성

아래 내용을 `system/skills/practice/daily-checkin/SKILL.md`에 저장합니다:

```markdown
---
name: daily-checkin
description: >
  팀원의 일일 체크인 문서를 생성합니다.
  "오늘 할 일 정리해줘", "데일리 체크인", "오늘 업무 계획",
  "아침 스탠드업" 같은 요청이 오면 이 스킬을 사용하세요.
---

# 일일 체크인 (Daily Check-in)

매일 아침 업무 시작 시 표준 형식으로 오늘의 계획을 정리합니다.

## 작업 절차

1. 현재 날짜와 요일을 확인한다
2. .fpof-state.json에서 현재 진행 중인 프로젝트를 확인한다
3. 최근 산출물(최근 3일)을 확인하여 이어서 할 작업을 파악한다
4. 아래 형식으로 체크인 문서를 생성한다

## 출력 형식

# 📋 데일리 체크인 — YYYY-MM-DD (요일)

## 🎯 오늘의 포커스
(가장 중요한 1가지)

## 📌 할 일 (To-Do)
- [ ] 작업 1 — 프로젝트명
- [ ] 작업 2 — 프로젝트명
- [ ] 작업 3 — 프로젝트명

## 🔄 이어서 할 것 (Carry-over)
(어제 미완료 → 오늘 이어서)

## 📊 참고 현황
- 현재 PDCA 단계: (상태)
- 진행 중 프로젝트: (목록)

## 예시

**Input:** "오늘 할 일 정리해줘"

**Output:**
# 📋 데일리 체크인 — 2026-03-25 (화)

## 🎯 오늘의 포커스
TRUSTED COMFORT 소재 벤치마크 최종 검토

## 📌 할 일
- [ ] TRUSTED COMFORT 소재 벤치마크 최종본 확인 — trusted-comfort/
- [ ] 성수 플래그십 오프닝 분석 리뷰 — retail-seongsu-flagship/
- [ ] w13 주간 대시보드 데이터 업데이트 — weekly/

## 🔄 이어서 할 것
- 웰니스 캡슐 컬렉션 인플루언서 콜라보 전략 (지난 세션에서 초안 완성)

## 📊 참고 현황
- 현재 PDCA 단계: Plan
- 진행 중 프로젝트: trusted-comfort, retail-seongsu-flagship
```

#### Step 3: 테스트

Claude Code에서 직접 테스트해 봅니다:

```
오늘 할 일 정리해줘
```

> 스킬이 트리거되어 표준 형식으로 체크인 문서가 만들어지는지 확인합니다.

## 5. FPOF 실무 스킬 아이디어

워크숍 이후 각자의 업무에 맞는 스킬을 만들어 보세요.

### 상품(P) 영역

| 스킬 아이디어 | 트리거 | 하는 일 |
|-------------|--------|---------|
| 사이즈 비율 계산기 | "사이즈 비율 잡아줘" | 카테고리별 표준 사이즈 비율 적용 + 지난 시즌 판매 데이터 기반 보정 |
| 소재 스펙 비교 | "소재 비교해줘" | 소재 A vs B 물성·단가·리드타임 비교표 생성 |
| QR 리오더 판단 | "리오더 해야 해?" | 판매 속도·재고·리드타임 기반 리오더 의사결정 보조 |

### 마케팅(M) 영역

| 스킬 아이디어 | 트리거 | 하는 일 |
|-------------|--------|---------|
| SNS 캡션 생성기 | "인스타 캡션 만들어줘" | 와키윌리 톤앤매너 + 채널별 톤 + 해시태그 자동 생성 |
| 인플루언서 브리프 | "씨딩 브리프 만들어줘" | 인플루언서 시딩용 표준 브리프 생성 (상품 정보·촬영 가이드·게시 조건) |
| 캠페인 ROI 계산 | "캠페인 효과 분석해줘" | 투입 비용 vs 매출·도달·전환 ROI 계산 + 벤치마크 비교 |

### 유통(D) 영역

| 스킬 아이디어 | 트리거 | 하는 일 |
|-------------|--------|---------|
| 매장별 실적 카드 | "매장 실적 카드 만들어줘" | 매장별 1-page 실적 요약 카드 (매출·객단가·전환율·재고) |
| VM 체크리스트 | "VM 점검해줘" | 사업부 가이드 기반 VM 체크리스트 자동 생성 (보기 좋게/집기 좋게/사기 좋게) |

### 공통 영역

| 스킬 아이디어 | 트리거 | 하는 일 |
|-------------|--------|---------|
| 회의록 → 액션아이템 | "회의 내용 정리해줘" | 회의록에서 결정사항·액션아이템·담당자·기한 자동 추출 |
| 보고서 포맷터 | "보고서 양식으로 바꿔줘" | 자유 형식 메모를 비케이브 표준 보고서 양식으로 변환 |

## 6. 스킬 Description 최적화

스킬을 만들었으면 description을 최적화하여 트리거 정확도를 높일 수 있습니다.

### 좋은 description vs 나쁜 description

**나쁜 예:**
```yaml
description: 데이터를 분석합니다.
```
→ 너무 넓음. 모든 데이터 관련 요청에 불필요하게 트리거됨.

**좋은 예:**
```yaml
description: >
  와키윌리 주간 매출 데이터(xlsx)를 읽고 경영진용 주간 리뷰 요약을 생성합니다.
  "주간 리뷰 만들어줘", "이번 주 실적 정리해줘", "매출 데이터 요약해줘",
  "경영진 보고용으로 정리" 같은 요청이 오면 반드시 이 스킬을 사용하세요.
  단순한 숫자 조회("매출 얼마야?")에는 사용하지 마세요.
```
→ 무엇을/언제/언제 안 되는지 명확.

### 최적화 팁

| 팁 | 설명 |
|----|------|
| **트리거 문구 나열** | 사용자가 실제로 말할 법한 문장을 3~5개 포함 |
| **부정 조건 명시** | "~에는 사용하지 마세요"로 오작동 방지 |
| **약간 적극적으로** | Claude는 스킬을 안 쓰는 쪽으로 치우치므로, 살짝 푸시하는 톤 |
| **구체적 파일/형식 언급** | "xlsx를 읽고", "마크다운으로 출력" 등 구체적 단서 |

## 7. 고급: 테스트 & 벤치마크

스킬이 복잡하거나 품질이 중요한 경우, 체계적으로 테스트할 수 있습니다.

### 테스트 파일 작성 (evals.json)

```json
{
  "skill_name": "my-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "실제 사용자가 할 법한 요청 1",
      "expected_output": "기대하는 결과 설명",
      "assertions": [
        {
          "description": "BTA 구성이 포함되어 있는가",
          "check": "output에 Basic/Trend/Accent 구분이 있어야 함"
        }
      ]
    }
  ]
}
```

### 테스트 실행 → 평가 → 개선 루프

```
1. 테스트 프롬프트 실행 (스킬 있음 vs 없음 비교)
2. 결과물 리뷰 (직접 확인 + 자동 검증)
3. 피드백 기반 스킬 수정
4. 다시 테스트
5. 만족할 때까지 반복
```

> 이 과정은 `/skill-creator` 명령어를 통해 자동화할 수 있습니다.

## 8. 스킬 공유하기

만든 스킬을 팀에 배포하는 방법:

### 방법 1: FPOF 프로젝트에 추가 (추천)

```bash
# 적절한 카테고리 폴더에 배치
cp -r my-skill/ system/skills/[카테고리]/my-skill/

# CLAUDE.md에 등록
# → 에이전시 테이블 또는 슬래시 명령어에 추가

# 커밋 & 푸시
git add system/skills/[카테고리]/my-skill/
git commit -m "feat: [스킬명] 스킬 추가"
git push
```

### 방법 2: .skill 파일로 패키징

```bash
# 패키징 (skill-creator 도구 사용)
python -m scripts.package_skill system/skills/[카테고리]/my-skill/

# → my-skill.skill 파일 생성
```
